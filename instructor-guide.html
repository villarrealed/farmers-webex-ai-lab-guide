WxCC Sandbox Setup

Set up a Webex Contact Center sandbox where 10 participants can access AI Agent Studio to build and test AI agents. Participants are builders — they create agents, not take calls.

Module 1: Create Your AI Agent ~15 min

Participants create a new Autonomous AI agent, name it, and write the system instructions that define its behavior.

Key Talking Points

"This is where you give your agent its personality and rules. Think of the system prompt as training a new employee — you're telling it exactly how to handle calls, what to ask for, and what it's allowed to do."
"The instructions we provide aren't just prompts — they define the agent's entire behavior boundaries. If you don't tell it to verify identity first, it won't. If you don't tell it to offer delivery options, it'll pick one arbitrarily."
"Notice we're writing instructions in natural language, not code. This is the power of LLM-based agents — you describe the behavior you want, and the AI figures out how to execute it."

What Participants Do

  • Access AI Agent Studio through Control Hub cross-launch
  • Create a new Autonomous agent (not Scripted)
  • Configure the agent profile (name, description)
  • Write system instructions using 3 copy blocks provided in the participant guide
Common issue: participants may try to access AI Agent Studio directly via URL instead of cross-launching from Control Hub. If they get "access denied," guide them through: Control Hub → Services → Contact Center → Digital → Webex Connect → App Tray → Webex AI Agent.

Module 2: Build Knowledge Base ~10 min

Participants create a knowledge base with pre-written articles and link it to their agent.

Key Talking Points

"The knowledge base is your agent's reference material — what it knows about your business without needing to call an API. Think FAQ answers, policy details, and procedures."
"In production, you'd connect to a document repository or CMS. For this lab, we pre-wrote three articles to ensure everyone gets consistent results."
"Notice the articles are written in a specific format — short paragraphs, clear headers, factual content. The better structured your knowledge base, the better your agent retrieves relevant information."

What Participants Do

  • Create a new knowledge base in AI Agent Studio
  • Add 3 pre-written articles (copy/paste from participant guide — copy buttons provided)
  • Link the knowledge base to their agent
Make sure participants name their knowledge base clearly (e.g., "Farmers Insurance KB"). If someone gets an error uploading, it's usually a formatting issue — have them re-copy the article text using the copy button (don't manually select text). Also remind them to link the KB to their agent after creating it — a common miss.

Module 3: Configure Actions ~10 min

Participants connect pre-configured MCP server actions to their agent and review the input entities.

Key Talking Points

"Actions are what make an AI agent actually useful. Without them, it can only talk — with actions, it can DO things. Verify identity, update records, send documents."
"Notice how the tool definitions came from the MCP server automatically — the agent studio didn't need manual input configuration. The tool name, description, and input parameters all sync from the server definition."
"This is the MCP standard at work — Model Context Protocol. It's an open standard that lets AI agents discover and use external tools without custom integrations for each one."

What Participants Do

  • Navigate to Actions tab → +Add Action → MCP Server
  • Find "edvill_Farmers_Insurance_Tools" and connect all 3 tools
  • Review input entities for each tool (entity tables in participant guide)
  • Understand the action flow: agent decides when to call → passes entities → receives result
If tools don't appear when participants click Add Action → MCP Server, this is a Control Hub provisioning issue on the admin side — not participant error. Double-check that the app status is "Allowed" and all tools are enabled. Participants cannot fix this themselves.

Module 4: Test & Validate ~15 min

Participants test their agent end-to-end using chat and voice preview, including edge cases.

Key Talking Points

"This is where it all comes together. The happy path tests the full flow: identify the customer → add their vehicle → deliver proof of insurance. All in one conversation."
"Try the edge cases too — what happens when you give a wrong date of birth? A good agent handles failures gracefully, explains what went wrong, and offers alternatives."
"If the agent doesn't behave as expected, go back to Module 1 and refine the instructions. This iterative process — test, observe, refine — is exactly how you'd develop a production agent."

Happy Path Test Data

Script for testing: "I'm Sarah Martinez. My date of birth is March 15, 1988. I just bought a 2024 Honda CR-V and need to add it to my policy. I use it for commuting. Can you send me proof of insurance by text?"

What Participants Do

  • Run the happy path via chat preview
  • Test voice preview (requires microphone permission)
  • Try edge cases: wrong DOB, invalid policy, missing information
  • Iterate on agent instructions if behavior isn't right
Voice preview requires browser microphone access — remind participants to allow the permission prompt. Some corporate environments or VPNs block WebRTC. If mic doesn't work, have them use chat preview instead (same functionality, just typed). Also: Render free tier cold starts may cause a 30-second delay on the first action call if the server was sleeping.

Module 5: AI Assistant Overview ~5 min

Conceptual wrap-up showing how AI Agent + AI Assistant create a complete solution.

Key Talking Points

"AI Agent handles the customer-facing side — it talks to customers 24/7. AI Assistant handles the human-agent side — when a call escalates, it gives your team full context, summaries, and suggested responses."
"They're complementary. In a full deployment, the AI Agent resolves what it can autonomously, and AI Assistant makes human agents more effective for everything else."

What Participants Do

  • Review the AI Agent vs AI Assistant comparison table
  • Understand the end-to-end customer journey flow
  • Complete the lab summary and discuss next steps
This is a wrap-up module — keep it brief (5 minutes max). The goal is to leave participants with the bigger picture. If running short on time, you can summarize verbally and skip having them read the detailed comparison. End with: "What questions do you have about deploying this in your environment?"

Troubleshooting

Tools don't appear in AI Agent Studio when adding actions
Check Control Hub → Apps → Agentic Apps. Ensure the app is set to "Allowed" and all tools are enabled in the Tools tab. Wait 1–2 minutes after provisioning for sync to complete.
MCP server health check fails (503 or timeout)
The Render free tier spins down after 15 minutes of inactivity. Visit https://farmers-insurance-mcp-server.onrender.com/health directly — it will cold start in ~30 seconds. Wait and retry.
Agent doesn't use the actions during testing
Check the system instructions (Module 1) — ensure they explicitly tell the agent to verify identity before adding vehicles. Also verify actions are connected to the agent (not just available in the catalog). Go to Actions tab and confirm all 3 show as connected.
Voice preview says "microphone access denied"
Browser needs mic permission. Click the lock/site-info icon in the address bar → allow microphone. Some corporate VPNs or security policies block WebRTC — try disconnecting VPN, or use chat preview as fallback.
Participant can't access AI Agent Studio
They need a Webex Connect login with AI Agent Studio entitlement. The correct path is: Control Hub → Services → Contact Center → Digital → Webex Connect → App Tray → Webex AI Agent. Direct URL access won't work without proper provisioning.
verify_policyholder_identity keeps returning "unable to verify"
Date of birth format must be exactly MM/DD/YYYY (e.g., 03/15/1988, not "March 15 1988"). The agent should convert natural language dates to this format, but if the system instructions aren't clear enough, the agent may pass the wrong format. Refine instructions to explicitly state the format conversion requirement.
First action call is very slow (30+ seconds)
This is Render free tier cold start. The MCP server sleeps after inactivity and takes ~30 seconds to wake up. Hit the health endpoint before the lab starts. In a production deployment, you'd use a paid tier with always-on instances.

Common Questions

"Can we use this with our real policy data?"

In production, you'd replace the static data in server.js with API calls to your policy management system. The MCP server architecture is identical — same tool definitions, same transport, same registration process. You just swap the data source behind the tools.

"How secure is this?"

For the lab, we use ApiKey auth for simplicity. Production deployments would use OAuth2 with client credentials for org-level service authentication. All traffic is HTTPS. The MCP server can implement any authentication scheme supported by the platform (OAuth2, custom headers, user tokens).

"Can the AI agent make mistakes?"

Yes — that's why the lab includes identity verification as the first step. In production, you'd add guardrails: human-in-the-loop approval for high-value transactions, audit logging for all actions taken, rate limiting, and configurable escalation thresholds.

"What happens if the MCP server goes down?"

The agent will still respond conversationally (using its instructions and knowledge base) but won't be able to complete transactions. A well-written system prompt tells the agent to gracefully inform the customer and offer to connect them with a human agent during business hours.

"How long to deploy something like this in production?"

The AI Agent Studio configuration itself takes hours. The significant effort is building production-grade MCP servers with real system integrations, comprehensive error handling, security review, and compliance approval — that's a development project ranging from weeks to months depending on the number of backend systems involved.

"Can we add more tools later?"

Absolutely. Add new tool definitions to the MCP server code, redeploy, and they'll automatically appear in Control Hub and AI Agent Studio. No re-registration needed — the platform discovers new tools from the server dynamically.

Data Flow for a Single Customer Interaction

  1. Customer speaks/types to the AI Agent via chat or voice preview
  2. Agent uses system instructions to determine conversation flow
  3. Agent consults knowledge base for factual answers (policy info, hours, coverage details)
  4. Agent calls verify_policyholder_identity with collected name + DOB
  5. On success → agent calls add_vehicle_to_policy with vehicle details
  6. On success → agent calls deliver_proof_of_insurance with delivery preference
  7. Agent confirms completion to customer with all reference numbers

Lab Reset Procedure

Steps to prepare the lab for a new cohort of participants:

  1. Verify MCP server health:
    curl https://farmers-insurance-mcp-server.onrender.com/health
  2. No data reset needed — the MCP server is stateless with hardcoded static data. Every call returns deterministic results regardless of previous usage.
  3. No participant cleanup needed — each participant creates their own agent in their own workspace. Previous participants' agents don't interfere.
  4. Wake up Render if sleeping: If the health check returns a 503 or times out, the free tier has spun down. Just visit the health URL — it'll cold start in ~30 seconds.
  5. Optional validation: Create a fresh test agent yourself and run the happy path to confirm the full flow works before participants arrive.
  6. Share the participant guide link:
    https://farmers-webex-ai-lab-guide.onrender.com/participant-guide.html
Pro tip: Open the health endpoint in a browser tab and leave it open during the lab. This keeps the Render service warm and prevents cold-start delays when participants first trigger an action.
