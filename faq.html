Getting Started & Login

How do I log in to Webex AI Agent Studio?

Navigate to Webex Control Hub and sign in with the lab credentials provided to you (check the credential bar at the bottom of the Participant Guide). From Control Hub, go to Services → AI Agent Studio to access the platform.

Tip: Use an incognito/private browser window to avoid conflicts with your personal Webex account.
I can't find AI Agent Studio in Control Hub. Where is it?

AI Agent Studio is located under Services in the left navigation of Webex Control Hub. If you don't see it:

  • Make sure you're logged in with the lab credentials (not your personal account)
  • Check that the URL shows the correct organization
  • Try refreshing the page or clearing your browser cache
Which browser should I use?

Google Chrome or Microsoft Edge (Chromium-based) are recommended. Safari and Firefox work but may have minor UI differences. Ensure your browser is up to date.

Can I use my personal Webex account for this lab?

No. The lab environment is pre-configured with specific permissions and MCP server access. Use only the lab credentials assigned to your seat. Your personal account won't have access to the lab organization's AI Agent Studio.

Agent Configuration

What's the difference between a Scripted and Autonomous agent?

Scripted agents follow a pre-defined conversation flow (like a decision tree). You design every possible path the conversation can take.

Autonomous agents (what we use in this lab) use AI/LLM to dynamically generate responses based on your instructions, knowledge base, and available actions. They can handle unexpected questions and adapt to natural conversation flow.

For this lab: We use an Autonomous agent because it's more flexible and better suited for natural customer interactions like adding a vehicle to a policy.
What goes in the Instructions field?

Instructions tell the agent how to behave. Think of it as the agent's "playbook." Include:

  • Persona — Who is the agent? (e.g., "You are a helpful Farmers Insurance virtual assistant")
  • Conversation flow — Steps the agent should follow (greet → verify identity → collect info → process → confirm)
  • Guardrails — What the agent should NOT do (e.g., "Never provide specific coverage pricing")
  • Tone & style — How the agent should sound (professional, friendly, concise)
  • Data formatting rules — How to handle dates, phone numbers, policy numbers
Important: Instructions work alongside your Knowledge Base. If they contradict each other, the Knowledge Base articles typically take priority. Keep them consistent.
Where do I set the Welcome Message?

The Welcome Message is configured in the Profile section of your agent. This is the first message the customer sees when a conversation starts. A good welcome message:

  • Identifies the agent ("Hi, I'm Farmers' virtual assistant")
  • Sets expectations ("I can help you add a vehicle to your policy")
  • Invites the customer to begin ("How can I help you today?")
How long can my Instructions be?

Instructions can be quite detailed — there's no strict character limit that you'll hit in this lab. However, best practices suggest keeping instructions:

  • Clear and structured — Use numbered steps and bullet points
  • Non-redundant — Don't repeat what's already in your Knowledge Base articles
  • Prioritized — Put the most important rules first

If your agent isn't following a specific instruction, it may be buried too deep. Move critical rules to the top.

Can I change the agent's name after creating it?

Yes. Go to your agent's Profile tab and update the name. Remember to publish after making changes for them to take effect.

Knowledge Base

What's the difference between Instructions and Knowledge Base?

Instructions = How the agent behaves (conversation flow, tone, rules)

Knowledge Base = What the agent knows (facts, policies, procedures, data)

Think of it this way: Instructions are like training a new employee on how to do their job. Knowledge Base is the reference manual they consult for specific information.

Example: Instruction says "Always verify identity before making changes." Knowledge article says "Verification requires matching at least one of: policy number, last name, or date of birth."
How do I add a Knowledge Base article?
  1. Go to Knowledge in your agent's configuration
  2. Click + Create an Article
  3. Give it a descriptive Source Name (e.g., "Vehicle Addition Process")
  4. Add a Description that helps the AI understand when to use this article
  5. Write the article content with clear, factual information
  6. Save and publish
My agent is giving wrong information. How do I fix it?

Wrong information usually comes from one of two places:

  1. Knowledge Base article has incorrect content — Edit the article with the correct information
  2. Instructions contradict the Knowledge Base — The agent gets confused when Instructions say one thing and an article says another. Make them consistent.
Key insight: When there's a conflict between Instructions and Knowledge Base, the Knowledge Base article typically wins. Always check both when debugging incorrect responses.
How many articles should I create?

For this lab, we use three focused articles. In production, the number depends on your use case. Best practices:

  • Keep articles focused — One topic per article (don't dump everything into one massive article)
  • Write clear descriptions — The AI uses the description to decide when to reference each article
  • Avoid duplication — If the same info appears in multiple articles, the agent may give inconsistent answers
Can I upload documents (PDF, Word) as knowledge?

Webex AI Agent Studio supports creating articles directly in the platform. For this lab, we write articles manually to maintain control over exactly what the agent knows. In production environments, you can integrate external knowledge sources depending on your deployment configuration.

Actions & MCP

What is MCP (Model Context Protocol)?

MCP is a standard protocol that allows AI agents to call external tools and services. In this lab, MCP connects your agent to a server that can:

  • Verify policyholder identity — Check customer info against records
  • Add a vehicle to a policy — Process the actual business transaction
  • Send proof of insurance — Deliver confirmation via SMS

The agent decides when to call these tools based on the conversation context and your Instructions.

How do I connect an MCP action to my agent?
  1. Go to Actions in your agent's configuration
  2. Browse Available Actions and select the MCP action
  3. Click Add to connect it
  4. The action's input entities (parameters it needs) will be displayed — review them to understand what data the agent needs to collect before calling the action
  5. Publish your agent
What are "input entities" for an action?

Input entities are the parameters that an action requires to execute. For example, the verify_policyholder action needs:

  • policy_number — The customer's policy number (e.g., "FRM-101")
  • last_name — Customer's last name
  • date_of_birth — Customer's date of birth

The agent will automatically collect these from the conversation before calling the action. The entity descriptions help the AI understand what format to expect and how to ask for each piece of information.

The MCP endpoint URL — what should it be?

For this lab, the MCP server endpoint is:

https://farmers-insurance-mcp-server.onrender.com/mcp

Common mistake: Make sure the URL ends in /mcp — not just the domain. The /mcp path is where the JSON-RPC protocol endpoint lives.
My action isn't firing during the conversation. Why?

Several possible causes:

  • Agent hasn't collected required entities — The agent won't call an action until it has all the required parameters. Check if it's still asking for information.
  • Instructions don't mention when to use the action — Add explicit guidance like "After verifying identity, use the add_vehicle action to process the request"
  • Action not connected — Go to Actions tab and confirm the action shows as connected
  • Agent not published — Changes only take effect after publishing
What happens if the MCP server is down?

If the MCP server is unreachable, the action will fail and the agent will receive an error response. How the agent handles this depends on your Instructions and error handling configuration. Good practice is to include guidance in your Instructions like: "If an action fails, apologize and offer to connect the customer with a human agent."

Note: The lab MCP server runs on Render.com's free tier and may take 30-60 seconds to "wake up" if it hasn't been used recently. If your first test fails, try again after a minute.

Testing & Preview

How do I test my agent?

Webex AI Agent Studio includes a built-in Preview panel on the right side of the screen. You can test via:

  • Chat mode — Type messages as if you were a customer
  • Voice mode — Click the microphone and speak naturally

The preview uses the published version of your agent. After making changes, you must publish before testing to see the updated behavior.

I made changes but the preview still shows old behavior. Why?

The most common reason: you haven't published yet. The preview always uses the last published version. After making any changes:

  1. Click Publish in the top-right area
  2. Wait for the publish confirmation
  3. Start a new preview session (the old session may still use cached context)
How do I start a fresh conversation in the preview?

Click the reset/new conversation button in the preview panel (usually an icon at the top of the chat). This clears all context from the previous conversation and starts from the welcome message.

Why this matters: If you're testing a new scenario but the agent remembers info from a previous test, results will be misleading. Always start fresh between test scenarios.
What test data should I use?

The lab MCP server has pre-loaded policyholder data. Use this for a successful verification:

  • Policy number: FRM-101
  • Last name: Smith
  • Date of birth: May 5, 1990 (any format works: 05/05/1990, May 5th 1990, etc.)

For a failed verification test, use any data that doesn't match (e.g., last name "Johnson", policy "555-000").

Can I receive a real SMS during testing?

Yes! The send_proof_of_insurance action delivers a real SMS to the phone number the agent collects during the conversation. When the agent asks where to send proof of insurance, provide your real phone number and you should receive an SMS within seconds.

Format: Provide a 10-digit US phone number. The agent handles formatting, but saying it clearly helps: "Send it to 555-123-4567"
The agent asked for my phone number but I never got an SMS. What happened?

Common causes:

  • Agent didn't pass the phone number to the action — Sometimes the agent collects the number but doesn't include it as a parameter when calling the action. Check Instructions to ensure the agent knows to pass the phone number.
  • Wrong number format — The system needs a valid 10-digit US number
  • SMS delivery delay — Wait 30-60 seconds; carrier delays happen
  • Action wasn't called — The agent may have acknowledged the request verbally without actually triggering the send action. Check the session details to verify.

Voice Interactions

How does Voice Preview work?

Voice Preview uses your device's microphone to capture speech, converts it to text (speech-to-text), processes it through the agent, and then speaks the response back (text-to-speech). It simulates what a phone-based customer interaction would sound like.

The agent misunderstood what I said. Why?

Speech-to-text isn't perfect. Common issues:

  • Names — "Smith" may be heard as "Smyth" or "Myth"
  • Numbers — "FRM-101" may be transcribed as "F R M 101" or "firm 101"
  • Dates — "May fifth, nineteen ninety" may become "may 5th 19 90"

The lab MCP server has built-in normalization to handle common speech-to-text variations. If verification still fails, try spelling it out clearly or switch to chat mode to confirm the issue is speech recognition vs. agent logic.

Agent responses sound too long in voice mode. How do I fix it?

Add a note to your Instructions like:

"For voice interactions, keep responses concise — no more than 2-3 sentences before pausing for the customer's reply. Avoid lists and bullet points in spoken responses."

Text that reads well in chat often sounds overwhelming when spoken aloud. The agent can follow voice-specific brevity rules if you include them in Instructions.

Can I listen to a recording of my voice test session?

Yes. Go to Sessions in the left navigation, find your voice session, click on it, and look for the audio player at the top of the transcript panel. You can play back the full conversation to hear exactly what was said and how the agent responded.

Troubleshooting

The agent says "I'm unable to verify your identity" even with correct data. Why?

The verification system uses flexible matching — any one matching field (policy number, last name, or date of birth) is enough to succeed. If all three fail, check:

  • Policy number format — Should be "FRM-101" but voice may produce "firm 101" or "FRM101". The server normalizes these, but extremely garbled input won't match.
  • Date format — The server accepts many formats (May 5 1990, 05/05/1990, May fifth nineteen ninety, etc.) but the agent must actually pass the date to the action.
  • Action parameters — Check the Sessions view → Actions Performed to see exactly what parameters were sent to the server.
The agent gives an empty or generic response after I provide all the info.

This usually means the MCP action returned an error or the agent couldn't interpret the response. Common fixes:

  • Check that the MCP server is awake — On Render.com free tier, the server sleeps after inactivity. Your first request wakes it up (takes 30-60 seconds). Try the conversation again.
  • Verify the action endpoint URL — Must be exactly: https://farmers-insurance-mcp-server.onrender.com/mcp
  • Check agent was published — Unpublished changes aren't reflected in conversations
The agent keeps asking the same question over and over.

This is usually an Instructions issue. The agent may not recognize that the customer already provided the information. Add to your Instructions:

"Track information the customer has already provided. Never ask for the same piece of information twice. If you already have their name, policy number, or date of birth from earlier in the conversation, use it."

The agent won't stop talking / gives very long responses.

Add explicit brevity rules to your Instructions:

  • "Keep responses under 3 sentences unless providing detailed confirmation"
  • "After asking a question, stop and wait for the customer's response"
  • "Don't list all the steps ahead — guide the customer one step at a time"
The agent is providing information I didn't put in the Knowledge Base.

Autonomous agents use a large language model (LLM) that has general knowledge. If you want the agent to only use your Knowledge Base for factual answers, add a guardrail to your Instructions:

"Only provide information about Farmers Insurance policies that is explicitly stated in your knowledge base. If you don't have specific information, say 'I don't have that information available — let me connect you with a specialist.'"

I accidentally deleted something. Can I undo it?

Check the History tab — it shows all published versions. While you can't directly "revert" to a previous version with one click, you can view the details of previous versions to see what the content was before your change, and manually restore it.

Sessions & Analytics

Where do I find past conversation sessions?

Click Sessions in the left navigation panel. You'll see a list of all conversations with your agent including session ID, start time, channel (chat/voice), and status indicators.

How do I see what the agent actually did during a conversation?
  1. Go to Sessions
  2. Click on the session you want to inspect
  3. Click Decrypt if prompted
  4. Expand the Actions Performed section — this shows every MCP tool call, the parameters sent, and the response received
  5. Expand the Slot Filling section to see what entities the agent extracted from the conversation
Tip: The Actions Performed section is the most useful for debugging. It shows you exactly what data was sent to the MCP server and what came back.
What do the session filters mean?
  • Error occurred — Sessions where an action failed or the agent hit an error
  • Agent handover — Sessions where the customer was transferred to a human
  • Test sessions — Sessions from the Preview panel (your lab testing)
  • Downvoted — Sessions where a user gave negative feedback
What's on the Analytics tab?

The Analytics tab shows high-level performance metrics over time:

  • Total Sessions — Number of conversations handled
  • Containment Rate — Percentage resolved without human escalation
  • Handover Rate — Percentage escalated to humans
  • Error Rate — Percentage that hit errors
  • Average Duration — Typical conversation length
  • User Satisfaction — Ratings/feedback from end users

Use the date range selector to focus on specific periods (e.g., after making a change).

What's on the History tab?

The History tab shows every published version of your agent's configuration — a chronological audit trail of all changes. Each entry shows:

  • Version number
  • Timestamp of the change
  • Who made the change
  • What was modified (instructions, knowledge, actions, profile, conversation settings)

Click any version to see the detailed diff. This is invaluable for tracking down "it worked yesterday" problems.

Publishing & Deployment

When do I need to publish?

Every time you make a change. Changes to Instructions, Knowledge Base articles, Actions, Profile, or Conversation settings only take effect after you publish. Think of it like saving a document — the preview and live agent always use the last published version.

Where is the Publish button?

The Publish button is in the upper-right area of the agent configuration page. After making changes, click Publish and wait for the confirmation before testing.

Is publishing instant?

Publishing is fast (usually a few seconds). However, if you have an active preview session, start a new one after publishing to ensure you're testing the latest version. Old sessions may still use cached context.

Can I break anything by publishing?

In this lab environment, no. You're working in an isolated lab organization — nothing you publish affects real customers or production systems. Experiment freely! If something breaks, check History tab to see what changed and fix it.

General Concepts

What is Webex AI Agent Studio?

Webex AI Agent Studio is a platform for building, testing, and deploying AI-powered virtual agents. It provides a no-code/low-code interface for creating agents that can handle customer interactions via chat and voice, powered by large language models (LLMs).

Key capabilities:

  • Natural language understanding and generation
  • Knowledge base for factual grounding
  • MCP actions to connect to external systems
  • Built-in testing (chat and voice preview)
  • Session analytics and conversation review
  • Integration with Webex Contact Center for human handoff
How does the agent decide what to say?

An autonomous agent uses a large language model (LLM) that considers:

  1. Your Instructions — The rules, persona, and flow you defined
  2. Knowledge Base — Articles it can reference for factual information
  3. Conversation context — Everything said so far in this conversation
  4. Available actions — Tools it knows it can call (and their descriptions)

The LLM synthesizes all of this to generate each response. That's why clear, consistent configuration across all four areas is important.

What happens when the agent can't help?

Depending on your configuration, the agent can:

  • Offer to transfer to a human agent — Configured via Conversation settings and Instructions
  • Gracefully decline — "I'm not able to help with that, but here's how to reach our support team..."
  • Ask for clarification — If it's unsure what the customer wants

In this lab, we configure the agent to offer human transfer after verification failures or out-of-scope requests.
